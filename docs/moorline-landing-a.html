System tide / control-home
Status proposal — on the human gate
Date 2026-06-25
§ 01

The system had a soul.

Every building system accumulates machinery — hooks, automations, pipelines, agents. But good systems have something beneath the machinery: a living definition of what is true, what is done, and how work is done here.

In this system, that thing is called CANON. Not a static document. Not a README that rots between edits. A living oracle improved with every cycle: do → define → refine. Work produces output; output feeds definition; definition tightens the next cycle. The loop is the method.

Canon is the answer to the question every new collaborator — human or agent — needs answered first: What is the current truth of this project? What have we learned? What do we commit to? It is the one thing that turns a pile of commits and decisions into an intelligible system.

"CANON is not a document about the system. It is the system — the crystallized residue of everything the work taught."

The challenge was always keeping canon central. Not as intention, but as structure. That challenge became a lesson the hard way.

§ 02

The long session.

It was the kind of session builders love: everything clicking, momentum building, each piece snapping into the next. The work was on execution machinery — the scaffolding that makes coordinated work possible at scale.

Git worktree isolation: every arc gets its own branch, its own directory, agents can't step on each other. Orca integration: the multi-agent runner wired to GitHub Issues and PRs, so every unit of work has an un-abandonable commitment. Automation hooks: role gates that enforce which agents can do which kinds of work. Pre-commit guards that block unsafe operations. An orchestrator edit-ban that physically prevented it from doing worker-work. Pull request flows with branch protection and required status checks.

Piece by piece, the machinery was becoming real. The agents were getting capable. The system was getting trustworthy. It felt like progress — because it was progress, on the machinery axis.

The trap

Execution machinery is rewarding to build. It produces visible, testable, demonstrable results. Every hook that trips, every gate that holds, every automation that fires feels like proof the system works.

The soul of the system — canon — is less legible. It doesn't fire hooks. It doesn't open PRs. It sits in a markdown file and waits.

And so, in the excitement of building the machinery, the makers got swept up. They were present, they were capable, they cared about quality — and yet, in that very session, something was silently left behind.

§ 03

The silence after.

When the session ended, CANON.md was broken. Not conspicuously broken — no crash, no error, no alarm. Silently broken.

The cannon-merge had run — or seemed to have run — but produced a corrupted result: top sections empty, structural content gone; duplicate journal entries, the same work recorded twice. The code had moved. The automation hooks were new. The Orca wiring was in. But canon had not absorbed any of it.

The concrete failure

Two things went wrong simultaneously. First: the merge itself was buggy — an idempotency gap let entries duplicate, and the atomic write wasn't truly atomic, leaving empty sections. Second: nobody updated canon after the changes. Not because they forgot to intend to — because the system had no mechanism that made "ship the changes without updating canon" impossible.

Reality had outrun canon. And the system had no way to even notice.

This is the failure mode that matters. Not the dramatic one — not the catastrophic data loss or the security breach — but the quiet one. The one where work ships, canon doesn't update, and the system's self-description silently diverges from its reality. Session by session. Week by week. Until canon is so out of date it's no longer consulted. At which point it has ceased to exist in any meaningful sense, even though the file is still there.

"The most dangerous failure isn't the one that errors — it's the one that succeeds quietly and leaves canon behind."

§ 04

The lesson already in the room.

The same session had solved an analogous problem. Earlier in the build, there was a different neglect: the orchestrator — the agent responsible for cross-project coordination — kept drifting into worker-work. Planning a feature while also implementing it. Reviewing while writing. The roles were collapsing.

First attempt: reminders. Add a note to the session prompt. It failed. When the work got interesting, the note was in the context window somewhere, unread.

Second attempt: willpower. Add it to the workflow checklist. "Before writing code, confirm you're in worker role." It failed. Willpower degrades under deadline pressure and accumulated context.

Third attempt: a mechanism. A PostToolUse hook that intercepted any code-writing action from the orchestrator role and blocked it with a non-negotiable exit code. The orchestrator physically could not do worker-work. Not because it remembered not to. Because the system made it technically impossible.

"Mechanism beat willpower. Not because willpower is weak — because mechanism is structural."

The lesson was already in the room. They had solved the problem of role collapse with a structural gate. The question was whether the same thinking could apply to the harder problem: keeping canon current. The answer became MOORLINE.

§ 05

How do you make it structurally impossible?

The question has a specific shape. Not "how do we remind ourselves to update canon" — reminders fail. Not "how do we build a checklist" — checklists are skipped under pressure. Not "how do we make it easier to update canon" — convenience isn't the blocker.

The question is: how do you make progress impossible without canon being current? How do you wire the system so there is no green path — no arc open, no arc close, no session, no PR merge, no automation — that leaves canon behind?

To answer it, four independent design lenses were opened simultaneously. Each was given the same brief: propose a model for canon-centrality. Each worked independently. Then each was subjected to adversarial critique with a single forcing question: where would canon still get neglected under your model?

The weaknesses exposed became the requirements. The strongest elements of each lens were extracted and synthesized. What emerged was MOORLINE.

§ 06

Today's checks ask the wrong question.

Before MOORLINE, the system had gates. They checked canon-consistency: does CANON.md's hash agree with its own stamps? Have the sections been committed? Is the format valid?

Those gates catch one failure mode: canon being internally inconsistent. But they miss the real one. The real failure is reality outrunning canon — code ships, systems change, and CANON.md's hash stays perfectly stable while becoming a description of something that no longer exists.

A consistent hash is not proof of freshness. It is proof of self-consistency — which is useless when the thing being self-consistently described has diverged from what the code actually does.

Wrong axis

"Does CANON.md's sha agree with its own stamps?"

Right axis

"Has canon absorbed the code that actually shipped?"

MOORLINE's keystone insight: the gate needs a second axis. Not just structural health (is canon internally valid) but reality-rev freshness: has canon absorbed what the code became? The two must be checked together, as a single oracle, or neither is sufficient.

§ 07

The model.

Core principle — MOORLINE

Canon is structurally central when the definition of work is the only door work can pass through — at birth, at every transition, and at exit — and that door is a single, two-axis, fail-loud oracle that no role, no interactivity, and no automation can route around.

The keystone is one predicate that fuses: structural health (canon is not rotted), two-axis freshness (canon has absorbed the code that shipped — not just that its own sha is consistent), substance (the delta encodes what the work taught, not a stub), and consultation (the arc was opened against canon).

tide cannon gate → (code: 0 | 1 | 2, reasons[])

0 = current    1 = stale    2 = oracle-error (fail-loud, never skip)

Every work transition — arc open, arc close, session start, Orca cron, PR merge — calls this one oracle. Mechanism, not willpower, beats discipline because there is no green path that leaves canon behind.

The model is six mechanisms. They compose. Each resolves a specific failure mode that adversarial critique revealed. Together they make the original neglect technically unreachable.

§ 08

Six mechanisms. One enforcement.

M1
tide cannon gate — the single two-axis oracle

One pure predicate, gate.decide(root) → (code, reasons), that is the definition of "canon is current." Tri-state: 0 = current, 1 = stale, 2 = oracle-error. Tri-state is load-bearing: callers must treat 2 as FAIL-LOUD, never skip — a broken oracle cannot silently disable enforcement.

Composes six clauses: no unmerged deltas; no open arc drifted on cannon-rev or reality-rev; lint passes; no dirty arc without substantive delta; no open arc with empty canon basis; no unpaid canon-debt. Backed by src/tide/gate.py, POSIX exit code, composes in shells, hooks, CI, and Orca --precheck. Every other gate delegates here.

Prevents The root neglect — nothing forced attention back to canon. Now no progress path is green while canon is stale, and a silently-dead gate is itself detected.
M2
Two-axis freshness: reality-rev

Every project's CANON.md carries a canon-covers: manifest — path globs it claims to describe. reality-rev = git-tree sha over those globs. tide arc new stamps both cannon-rev and reality-rev; close re-stamps.

The gate reports STALE when reality-rev moved but the merged delta touched no canonical section (## What it is / ## State & components / ## Interfaces) — meaning "code shipped, canon didn't." A configurable commit/time threshold turns "canon lags reality" from an unobservable state into a tripwire.

Prevents The exact original failure — "forgot to update canon after changes." Bumping the rev with a trivial delta no longer clears the gate while the top sections rot.
M3
DEFINE as a first-class lifecycle state

Insert DEFINE into the contract state machine: new → sign → running → output → DEFINE → accept → close. No output → close edge exists.

DEFINE's exit criterion: a delta that routes content into ≥1 canonical section (journal-only = worklog, not refinement) OR a signed noop (--noop --reason), which becomes auditable canon-debt that re-stales the gate until reconciled. Symmetrically, tide arc new pre-fills a canon basis block quoting the relevant CANON.md sections; an open arc with empty basis is STALE.

The -f force escape is not keyed on interactivity — any override writes a gate-blocking canon-debt entry. Overrides convert neglect into visible, payable debt. Never a clean exit.

Prevents Delta-shaped tokens that clear a presence-check; canon being write-only and never consulted; "trusted interactive human types close -f."
M4
Atomic, serialized, idempotent, trunk-only merge

tide cannon merge is the single serialization point, safe under Orca parallelism. Idempotent: reads the delta's merged: flag first, no-op if already merged — the actual cause-fix for the duplicate-journal corruption, not a lint patch papering the symptom.

Locked + atomic: flock on CANON.md for read-modify-write, write-temp + os.replace, compare-and-swap on cannon-rev (re-read base under lock, abort if base moved). Self-linting: build merged text, run cannon lint, swap only on pass. Trunk-only: refuses to run inside an Orca worktree. Workers write delta.md; the orchestrator merges once, on the integration branch.

Prevents The literal corruption observed (empty top sections + duplicate journal), lost-update races between concurrent automations, and the silent multi-worktree defeat of single-serialization.
M5
Write-time + read-time guardians

(a) canon-pulse: session start reordered so the canon block (rev, updated:, lint verdict, lag count, current-pointer) is the first text of every session, healthy or not.

(b) canon-lint: pure structural invariant — canonical sections non-empty for maintained projects, unique top headings, dedup'd append-only journal, no template placeholders. Substance floor distinguishes "seeded-empty new project" from "hollow maintained canon."

(c) PostToolUse hook on Edit/Write whose file_path resolves to CANON.md: runs lint, blocks (exit 2) on bad write. Pre-commit hook forbids hand-edits: any CANON.md diff not produced by tide cannon merge is rejected. Canon becomes append-only-via-merge.

Prevents Canon out of sight at entry; a corrupting hand-edit sitting broken until the next transition; "quickly fix the empty sections by hand" with zero provenance.
M6
Meta / factory canon is arc-shaped too

The neglect happened in control-home / factory canon — meta-work that no project arc touches. So the tide repo and control-home are first-class projects with their own CANON.md, arcs, contracts, and revs.

A repo-level pre-push / CI gate tide cannon gate --home runs on ANY change to tide's own code or control-home and blocks the merge unless the home canon's delta is reconciled. No "it's just config/infra" escape hatch — that hatch is precisely what failed.

Prevents The exact surface that broke being structurally outside every gate — meta-work silently excluded from the enforcement it helped build.
§ 09

How it rides Orca-native execution.

MOORLINE is designed to compose with Orca's native execution model, not fight it. One arc ↔ one worktree ↔ one GitHub issue ↔ one PR ↔ one contract ↔ one cannon-delta, all keyed by both cannon-rev and reality-rev stamped at birth into arc frontmatter, the issue body, and a committed .tide/state/arc-rev.

bind-at-birth → tide arc new --issue is the only entrypoint; it calls the gate BEFORE any gh/worktree side-effect. An owed delta means no issue and no worktree exist.
enforce-at-exit → tide cannon gate as a required PR status check independently re-derives the arc↔issue↔rev binding from the PR branch and fails any PR whose worktree was not born through tide arc new. Un-stamped work can commit but never reach a terminal state.
precheck-escalates → Every tide-managed Orca automation is created with --precheck 'tide cannon gate --project <p>'. On canon-debt it opens a tracked issue, pushes a notification, and after N aborted cycles auto-spawns a dedicated reconcile-canon arc. Precheck escalates — never silently stalls.
terminal-binding → The only path that closes the issue / merges the PR / sets status: completed is tide arc close, which runs trunk-only cannon merge first and bumps both revs. Branch protection has include-administrators ON plus a server-side pre-receive ruleset — even the solo owner's late-night force-merge runs the gate.

tide stays the method (canon · arc · contract · two revs · DEFINE · drift). Orca stays the mechanism (worktree · agent launch · board · PR · cron). The two revs carried across both surfaces are the single binding that makes neglect technically impossible, not a remembered discipline.

§ 10

How it would have prevented this.

The concrete moment: humans and agents got excited about Orca/worktree/issue/PR machinery, left cannon-merge broken, forgot to update canon. Here are the stop points MOORLINE would have enforced, in order:

1
M6 — no meta-work escapes the arc
The Orca/hooks/PR work touched tide's own code and control-home. M6 makes that arc-shaped: the pre-push gate would have refused to merge the execution-machinery changes until the home canon's delta was reconciled. The work could not have "shipped past" canon.
2
M2 + M1 + M5 — reality moved, gate trips, banner shows
Reality moved (lots of new code) while CANON.md's top sections stayed empty. M2's reality-rev trips STALE, M1 surfaces it as non-zero on the next arc-open, and M5 canon-pulse makes it the first text on screen at session start.
3
M4 + M5 — the corruption is non-representable
Empty top sections and duplicate journal entries could not have been a resting state: M4's idempotent + locked + self-linting merge refuses to produce duplicates or swap in a file that fails lint; M5's PostToolUse + pre-commit guardians reject the corrupting write at the instant it happens.
4
M3 — no green path under pressure
Even under deadline pressure, closing with a stub or a -f would not clear the gate. M3 demands a canonical-section delta or a signed noop that itself becomes gate-blocking canon-debt. There was no green path that left canon behind.
§ 11

The fastest path to canon-central.

Six steps, sequenced by leverage. Each step unblocks the next. The first two are highest-leverage: they fix the confirmed bug and establish the oracle everything else calls.

1
cannon lint + idempotent/locked/atomic cannon merge (M4 + M5b)
Cause-fix the confirmed duplicate-journal bug. Make corruption non-representable. Smallest, highest-leverage, unblocks everything. Add per-entry content-hash dedup and flock + os.replace + CAS.
2
tide cannon gate tri-state oracle (M1)
One gate.py, POSIX exit code. The keystone every later call site reuses. Clauses for lint + unmerged + drift.
3
reality-rev two-axis freshness (M2)
Add canon-covers: manifest + reality-rev stamping; wire into the gate. This is what actually fixes the original neglect direction.
4
DEFINE state + substance/consultation guard (M3)
Make the loop's front and back halves structural. Non-interactive-proof force-debt. The lifecycle becomes: new → sign → running → output → DEFINE → accept → close.
5
Hooks: canon-pulse + PostToolUse guardian + pre-commit ban (M5a/M5c)
Cheap, high-visibility, closes the write-time hole. The same mechanism that beat willpower for the orchestrator edit-ban, applied to canon's own file.
6
Orca wiring + M6 home-as-project gate
Required PR check that re-derives the binding. Automation precheck-with-escalation. Terminal-state binding. include-administrators ON + pre-receive. Last because it depends on the oracle (step 2) being trustworthy.

◆ ◆ ◆

Mechanism, not willpower.
That is MOORLINE.

The makers were present. They were capable. They cared. And canon still got neglected — because presence and intention are not mechanisms. They degrade under pressure, context, and excitement.

MOORLINE doesn't ask for more discipline. It removes the path that requires it. Every work transition calls the same oracle. The oracle enforces both axes. The gate has no interactive bypass that leaves canon behind.

The system has a soul. MOORLINE makes sure it stays at the center.
